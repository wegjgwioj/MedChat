# 面向患者的智能医疗伴诊与问答 Agent 重构方案与当前落地现状

## 1. 文档定位

本文档不再把“目标架构”和“当前仓库已实现能力”混写。  
当前版本的定位是：

- 说明项目为什么要重构
- 说明目标架构想达到什么水平
- 明确哪些已经实现、哪些只实现了最小可用版、哪些仍未完成

如果后续代码继续演进，应优先更新本文档中的“当前状态”部分，避免再次出现“文档写了但仓库里没有”的情况。

------

## 2. 项目背景与核心痛点

在面向患者的医疗问答场景中，通用 LLM 即使有较强的对话能力，在严肃医疗 RAG 架构下仍然容易遇到以下问题：

1. **语义鸿沟**

   患者输入往往很短、很模糊，例如“胃痛”“头晕”“发烧”。  
   如果直接把这类 query 拿去做一次性检索，通常会召回大量宽泛、噪声较高的证据，导致回答看似完整，但引用依据并不稳定。

2. **长文本与 OCR 噪声**

   医疗知识库、报告文本和 OCR 结果都可能很长。  
   如果切分策略太粗暴，关键医学实体会被截断；如果 OCR 文本噪声太高，又会进一步污染召回结果。

3. **多轮记忆与安全约束**

   多轮问诊中，系统需要记住患者症状、病史、过敏史等上下文。  
   一旦回答与既往病史冲突，或者系统忘记了前面对话中的关键信息，就会带来明显的医疗风险。

4. **隐私与合规**

   患者输入可能包含手机号、地址、身份证、就诊信息等隐私内容。  
   这些信息如果直接进入日志、持久化或 LLM 调用链路，会带来合规与安全问题。

------

## 3. 目标架构

目标上，本项目希望演进为三层能力协同的医疗 Agent：

### 3.1 检索层

- 支持医疗长文本的稳健切分
- 支持稀疏检索 + 稠密检索 + rerank 的混合召回
- 支持低质量证据过滤
- 支持 OCR 文本入库与检索

### 3.2 问诊 Agent 层

- 支持主动追问，而不是单轮直答
- 支持结构化槽位补全
- 支持知识问答与个体问诊分流
- 支持红旗症状升级提示

### 3.3 安全与记忆层

- 支持短期会话上下文
- 支持纵向病史/过敏史沉淀
- 支持回答与病史冲突检测
- 支持隐私脱敏与日志最小化

------

## 4. 当前实现状态

下面这部分只描述当前仓库中已经存在的实现。

### 4.1 已完成

1. **多轮 Agent 状态机**

   已实现 `Ask / Answer / Escalate` 三种模式，主链路包含：

   - SafetyGate
   - MemoryUpdate
   - TriagePlanner
   - RAGRetrieve
   - AnswerCompose
   - PersistState

2. **主动追问与槽位补全**

   已支持基于槽位缺口的结构化追问，能够生成：

   - `ask_text`
   - `questions`
   - `next_questions`

3. **红旗症状分流**

   已支持对胸痛、呼吸困难、意识异常等高风险描述进行升级提示，不继续普通问诊流程。

4. **RAG 混合检索基本链路**

   当前已实现：

   - 稀疏检索
   - 稠密检索
   - rerank
   - 阈值过滤
   - 证据契约统一输出

5. **OCR 接入链路**

   当前已支持：

   - OCR 任务创建
   - OCR 状态查询
   - OCR 文本提取
   - OCR 结果自动入库

6. **纵向档案的最小可用版**

   当前已支持从对话中抽取并沉淀最小粒度的长期记录，用于后续安全判断和摘要构建。

7. **基础评测与测试闭环**

   当前仓库已经包含：

   - pytest 单测
   - Agent 行为测试
   - RAG 检索测试
   - OCR 测试
   - 一组基础评测脚本

### 4.2 部分完成

1. **隐私保护**

   当前已具备部分脱敏与日志最小化能力，但还不能称为“全链路、强合规”的完整 PII 方案。

2. **短期记忆**

   当前已有会话状态与上下文维护能力，但实现形态和最初方案中的理想版本并不完全一致。

3. **安全熔断**

   当前已有病史/回答冲突判断能力，但还没有完全达到“独立轻量 NLI/SLM 裁判模型”的成熟形态。

4. **流式响应**

   当前已提供 SSE 接口与前端联调能力，但仍属于工程可用版，不代表完整的生产级流式推理链路。

5. **缓存能力**

   当前已具备缓存相关实现，但是否作为稳定线上能力启用，仍需结合部署环境、Redis 可用性和回退策略进一步收口。

### 4.3 未完成或尚未形成稳定能力

1. **完整的 NER 医疗信息抽取体系**

   当前长期记录以规则和最小可用实现为主，还没有形成成熟的医疗 NER 管线。

2. **完整的 NLI / DeBERTa 小模型安全裁判**

   这是目标方向，但当前仓库还没有形成完全稳定、可独立部署的版本。

3. **严格的评测闭环**

   当前尚未建立完整的：

   - 700 条 Self-Instruct 医疗测试集
   - RAGAS 评估闭环
   - Physician-in-the-loop 正式抽检流程

4. **文档里曾经写过的最终指标**

   例如：

   - Recall@30 = 84%
   - Faithfulness = 85%
   - 拦截率 98.2%

   当前仓库没有与这些数字一一对应的、完整可复现的评测证据，因此这些数字不能再被视为“当前已达成事实”。

------

## 5. 当前真实达成度判断

如果从工程角度评估当前项目状态，我会给出以下判断：

- **主骨架已完成**
  现在系统已经不再是单纯原型，核心链路、测试、前后端联动、OCR、Agent 编排都已经成形。

- **方案级目标部分达成**
  与理想中的“工业级、强合规、强评测医疗 Agent”相比，当前版本更接近“高完成度 MVP + 一部分增强能力”。

- **总体达成度约 60%~70%**
  这个判断主要基于：
  - 主链路已落地
  - 关键工程能力已具备
  - 但安全、评测、合规和指标闭环仍未完全收口

------

## 6. 后续优先级建议

如果继续重构，建议优先级如下：

1. **先收口安全与隐私**

   - 明确 PII 脱敏范围
   - 明确日志策略
   - 明确长期记录的入库边界

2. **再收口安全判断链路**

   - 区分规则安全、LLM 安全裁判、小模型安全裁判
   - 把“当前版本”和“目标版本”在代码与文档里分清楚

3. **最后补评测证据**

   - 建立可复现测试集
   - 固定评测脚本
   - 再填写最终指标
